# Agent 架构选型决策地图：何时用普通并发、Dynamic Workflow，还是 Agent Team？

## 🔍 三者本质：不是“口味”，而是三种范式

**Jen:**  
很多人以为 Ultra Code、Dynamic Workflow 和 Agent Team 是同一技术的三种使用方式——其实完全不是。它们解决的是三类根本不同的问题，对应三种正交的设计维度。

**Mark:**  
对。我们先拆开看：  
- **Ultra Code** 是一个全局推理增强开关。它不改变架构，但提升模型调用强度，并让 Cloud 智能判断——哪些任务该升维到 Workflow 级别处理；  
- **Dynamic Workflow** 是 Cloud 实时生成的轻量脚本，用代码编排一批**隔离执行**的 Agent。它适合确定性高、可缓存、需复用结果的任务；  
- **Agent Team** 则是一组有明确身份（如 “前端工程师”“后端架构师”“DBA”）的协作体，靠共享任务列表和**点对点实时消息**推进工作。

**John:**  
所以核心差异只有一句：**Agent 之间能不能说话？**  
- 在 Agent Team 中，前端发现接口定义不符，能立刻私聊后端：“这个字段类型要改，我等你更新再继续”；  
- 而 Dynamic Workflow 的每个 Agent 都是“单向流水线”中的一环——结构在脚本生成时就固化，只能把上一阶段输出作为下一阶段输入，无法动态协商、推翻约定或中途调整。

## ⚙️ 执行逻辑对比：弹性 vs 确定性

**Jen:**  
从驱动机制和扩展性来看，差异非常清晰：  

| 维度 | Agent Team | Dynamic Workflow |
|------|------------|------------------|
| **驱动方式** | LM 主导，逐轮对话推进，天然支持反思与修正 | 代码驱动，结构固定，执行路径在生成时即确定 |
| **规模上限** | 通常 2–5 个队友为佳；人数增加会导致消息漂移、共识衰减、结果不可复现 | 支持最多 16 并发 Agent，生命周期可覆盖 1000+ 任务单元 |
| **成本特征** | 基准成本的 2–4 倍（含多轮协商、上下文维持、角色状态同步） | 成本主要取决于数据量，但有明确预算上限（Cloud 可预估并拦截超支） |

**Mark:**  
简单说：  
- **Team 买的是“协商弹性”**——当契约未定、接口尚在演进、组件必须高频对齐时，它是唯一选择；  
- **Workflow 买的是“规模复现性”与“验证结构”**——当你需要批量跑通 200 个 API 的响应一致性，或从 5000 条日志中筛选异常模式，它才是最优解。

## 🎯 场景匹配指南：四步精准决策法

**John:**  
真正落地时，我们推荐按这四步判断：

**第一步：是否只是几个互不依赖的小任务？**  
→ 直接用**普通并发 Agent**（零编排、低开销、秒级启动）。

**第二步：是否涉及大量独立单元 + 验证密集 + 规模未知 + 排序/筛选需求？**  
→ 优先选**Dynamic Workflow**。典型场景包括：未知数量的文件格式校验、海量候选集的对抗过滤、搜索结果的多维打分排序。

**第三步：是否只有 2–5 个强依赖组件，且接口仍在快速迭代？**  
→ 这才是**Agent Team 的黄金场景**。例如：数据库 Schema、API 契约、前端 UI 三者同步设计，任一环节变更都需即时通知关联方协同调整。

**第四步：整段会话是否属于“重体力活”？**  
→ 如审计报告生成、系统迁移验证、全量数据一致性批量检查……此时应开启**Ultra Code 全局策略**，让 Cloud 自动将关键子任务升维至 Workflow 级处理，兼顾深度与效率。

## 🛠️ Dynamic Workflow 的六种可靠模式

**Jen:**  
Dynamic Workflow 不是自由发挥——它的可靠性，来自六种经过验证的结构化模式。当你能准确描述需求，Cloud 生成的 harness（执行框架）将远比一句“多派几个 Agent”更精准、更可控：

- **分类分发（Classify & Route）**：同一操作需按文件类型、数据源或业务域分流处理；  
- **删除合并（Delete & Merge）**：将多个小步骤并行执行，再聚合结果（如并发解析 10 份配置，统一生成部署清单）；  
- **对抗验证（Adversarial Verification）**：为每个发现自动配一个独立 “Skeptic Agent”，交叉质疑、反向验证；  
- **生成过滤（Generate & Filter）**：先批量产出候选方案（如 200 条 SQL），再去重、降噪、按规则筛出 Top 5；  
- **锦标赛（Tournament）**：多个方案两两比对，逐轮淘汰，最终收敛最优解；  
- **循环制收敛（Iterative Convergence）**：持续运行直到连续 N 轮无新问题发现，确保穷尽性。

**Mark:**  
记住：能说出这些模式，等于给了 Cloud 一张精确的“施工图”。它不再猜测你要什么，而是直接构建可验证、可审计、可复现的工作流。

## ✅ 最终一句话决策口诀

**John:**  
把整张选型地图压缩成一句可立即行动的口诀：  
> **几个独立任务 → 普通并发；  
> 大量条目 / 规模验证 / 排序筛选 → Dynamic Workflow；  
> 少数几个彼此依赖、接口尚在变化的组件 → Agent Team；  
> 整段会话像审计、迁移、批量验证这类“重活” → 打开 Ultra Code 全局增强。**  

架构没有银弹，但有清晰的地图——选对范式，80% 的协同熵增和资源浪费，从一开始就被规避了。
